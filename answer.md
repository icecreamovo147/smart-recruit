# 拓展设计方案思考题

> 题目：如果需要将本系统对接集成到 OpenClaw 或 Hermes 等三方 AI Agent 平台中，请详细写出完整接入实施方案（仅设计思路与流程，不需要代码实现）。

---

## 一、背景理解

OpenClaw、Hermes 等三方 AI Agent 平台的核心能力是：让 Agent 能够通过**工具调用（Tool Use / Function Calling）**与外部系统产生真实交互。本系统需要将"智能招聘"领域的业务能力，封装为标准工具，暴露给 Agent 平台进行编排和调用。

---

## 二、整体接入架构

```
三方 Agent 平台（OpenClaw / Hermes）
        │
        │  工具调用请求（Tool Call）
        ▼
  招聘系统 MCP / Plugin 适配层
        │
        │  内部 gRPC 调用
        ▼
  Logic gRPC 核心业务服务
        │
   ┌────┴────┐
 MySQL      OSS
```

接入的核心思路是：**在现有 Logic 服务之上，新增一层"能力适配层"**，将业务能力描述为工具定义，供 Agent 平台发现和调用，不改动原有核心服务。

---

## 三、接入实施方案

### 步骤一：梳理并定义可暴露的工具能力

首先需要盘点本系统中，哪些业务能力适合作为工具（Tool）暴露给 Agent：

| 工具名称 | 功能说明 | 输入参数 | 返回结果 |
|----------|----------|----------|----------|
| `query_total_applications` | 查询全平台投递总数 | 无 / 时间范围 | 投递总人数 |
| `query_job_applications` | 查询指定岗位投递数 | job_id 或岗位名称 | 该岗位投递人数 |
| `search_candidates` | 按条件筛选候选人 | 学历、技能标签、岗位 | 候选人列表 |
| `get_job_heat_ranking` | 查询岗位热度排行 | top_n | 岗位热度排名 |
| `get_job_list` | 获取当前招募中的岗位列表 | 分页参数 | 岗位列表 |
| `get_candidate_profile` | 获取指定候选人详情 | user_id | 候选人档案结构 |

工具定义需要包含：**名称、描述（自然语言说明供 Agent 理解）、参数 schema（JSON Schema 格式）、返回格式说明**。

---

### 步骤二：选择接入协议标准

根据目标平台支持的协议，选择一种标准方式暴露工具：

**方案 A：MCP（Model Context Protocol）协议接入**

适用于支持 MCP 标准的平台（如 OpenClaw 等兼容 MCP 的 Agent 框架）：

- 实现一个 **MCP Server**，通过 JSON-RPC over HTTP/SSE 暴露工具列表
- MCP Server 作为适配层，接收 Agent 平台的 `tools/call` 请求，转发给 Logic gRPC 服务执行
- 返回结构化结果给 Agent 平台继续推理

**方案 B：OpenAPI / Plugin 描述文件接入**

适用于支持 OpenAPI Schema 或 ChatGPT Plugin 规范的平台：

- 编写标准 `openapi.yaml`，描述每个工具对应的 HTTP 接口
- Agent 平台通过 HTTP 调用对应接口，接口内部调用 Logic gRPC 服务

两种方案可同时实现，对外分别暴露，提升平台兼容性。

---

### 步骤三：构建适配层服务

新增一个独立的**适配层服务（Adapter Service）**，职责是：

1. **维护工具注册表**：管理所有工具的名称、描述、参数 schema，支持动态注册
2. **接收 Agent 调用请求**：解析平台传入的工具名和参数
3. **鉴权与频控**：验证调用方的 API Key，防止未授权访问和滥用
4. **路由执行**：根据工具名，调用对应的 Logic gRPC 接口执行业务逻辑
5. **结果格式化**：将 gRPC 返回的结构化数据，序列化为 Agent 平台期望的文本/JSON 格式

适配层与 Logic 服务之间仍然通过 gRPC 调用，保持原有微服务分层架构不变。

---

### 步骤四：在 Agent 平台侧完成注册配置

在 OpenClaw / Hermes 平台控制台进行以下配置：

1. **注册工具 / 插件**：上传工具定义（JSON Schema 描述文件）或填写 MCP Server 地址
2. **配置访问凭证**：填写招聘系统适配层颁发的 API Key，平台调用时携带于请求头
3. **配置系统提示词（System Prompt）**：向 Agent 描述本系统的业务背景，如"你是一个招聘数据助手，可以查询岗位投递情况、筛选候选人"，帮助 Agent 更准确理解何时使用哪个工具
4. **测试联调**：在平台沙箱中发送测试问题，验证 Agent 能否正确识别意图、调用工具、解释结果

---

### 步骤五：安全与权限管控设计

接入三方平台后，安全性是重点：

- **API Key 隔离**：为每个接入的第三方平台单独生成 API Key，支持随时吊销
- **权限最小化原则**：适配层对外只开放查询类工具，写操作（投递、修改档案等）不对 Agent 暴露，防止 Agent 产生意外副作用
- **敏感数据脱敏**：候选人手机号、邮箱等在返回给 Agent 时进行脱敏处理（如 `138****8888`），防止数据泄露
- **调用频率限制**：对每个 API Key 设置每分钟/每小时最大调用次数，防止 Agent 进入死循环或被恶意滥用
- **调用日志审计**：所有来自三方平台的工具调用记录完整日志，包括调用时间、工具名、输入参数、执行结果，便于排查问题

---

### 步骤六：持续迭代与监控

- **工具版本管理**：当业务变化需要修改工具定义时，通过版本号平滑升级，避免影响已接入平台
- **调用质量监控**：统计各工具的调用次数、失败率、响应时延，发现问题及时优化
- **工具描述优化**：根据 Agent 的实际调用准确率，不断调整工具的自然语言描述，提升 Agent 理解意图的精准度

---

## 四、总结

本方案的核心思路是**新增适配层，不改动原有服务**：

```
原有系统架构不变
    ↓
新增 MCP/Plugin 适配层（工具注册 + 路由转发）
    ↓
三方 Agent 平台通过标准协议调用适配层
    ↓
适配层内部通过 gRPC 调用已有 Logic 服务
```

这样既保障了原有系统的稳定性，又实现了对 Agent 平台的标准化开放，同时通过适配层集中处理安全、权限、频控等横切关注点，架构清晰，易于维护和扩展。
